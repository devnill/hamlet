# WI-115: Village initialization — seed initial structures on village creation

## Context

`WorldStateManager.add_work_units()` at `manager.py:465-557` searches the agent's village for the nearest structure of the required `StructureType`. If none exists, it logs at DEBUG and returns without accumulating. On a fresh world, villages have zero structures. Every tool call's work units are silently dropped, and the structure-building mechanic (GP-8) is permanently broken until an expansion event triggers road building (which itself is only triggered after the agent count exceeds `expansion_threshold`).

The fix: seed initial structures when a village is first created.

## Which StructureTypes to seed

The `TOOL_TO_STRUCTURE` mapping (currently at `engine.py:28-35`, moved to `world_state/rules.py` by WI-117) maps tools to:
- `StructureType.LIBRARY` (for read/search tools)
- `StructureType.WORKSHOP` (for write/edit tools)
- `StructureType.FORGE` (for bash/execute tools)

Seed one of each: LIBRARY, WORKSHOP, FORGE. This ensures any tool call type can immediately contribute work units.

## Where to add seeding

### Location 1: get_or_create_project (manager.py:134)

This is the correct canonical location. When a new project is created (line where `Village` is first constructed, around manager.py:150-159), immediately after adding the village to the world state, seed the three initial structures.

```python
# After village is created and added:
await self._seed_initial_structures(village)
```

### Location 2: get_or_create_agent fallback (manager.py:274-287)

This fallback creates a placeholder village when no project entity exists (which happens when get_or_create_project is never called — fixed by WI-114, but defensive). After WI-114 this path should be unreachable in practice, but seed structures here too for safety.

### _seed_initial_structures helper

```python
async def _seed_initial_structures(self, village: Village) -> None:
    """Create one LIBRARY, WORKSHOP, and FORGE near the village center."""
    from .types import StructureType  # avoid circular if needed
    center = village.center
    # Place structures adjacent to center, avoiding occupied positions
    offsets = [(1, 0), (-1, 0), (0, 1), (0, -1), (2, 0), (-2, 0), (0, 2), (0, -2)]
    types_to_seed = [StructureType.LIBRARY, StructureType.WORKSHOP, StructureType.FORGE]
    placed = 0
    for dx, dy in offsets:
        if placed >= len(types_to_seed):
            break
        pos = Position(x=center.x + dx, y=center.y + dy)
        if not self._grid.is_occupied(pos):
            structure_type = types_to_seed[placed]
            await self.create_structure(
                village_id=village.id,
                structure_type=structure_type,
                position=pos,
            )
            placed += 1
```

## Key locations

- `manager.py:134` — `get_or_create_project` method start
- `manager.py:150-159` — village creation inside get_or_create_project
- `manager.py:274-287` — placeholder village creation in get_or_create_agent
- `manager.py:511-521` — the silent-drop in add_work_units (no change needed here; it works correctly once structures exist)
- `world_state/types.py` — `StructureType` enum values (LIBRARY, WORKSHOP, FORGE)
- `manager.py` — `create_structure` method (use existing implementation)

## Verification

After this fix, on a fresh world:
1. First hook event creates a project → village is founded with LIBRARY, WORKSHOP, FORGE
2. Subsequent tool calls call add_work_units → structures are found → work_units accumulates
3. At threshold, structure advances from stage 0 → 1 (foundation → frame)
