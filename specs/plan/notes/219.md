# WI-219 Implementation Notes

## Context

`WorldStateManager.found_village` creates a new `Village` record, adds it to `_state.villages`, and queues a persistence write. It does not call `_seed_initial_structures`. Compare with `get_or_create_project` (line 226 of `manager.py`), which creates a village and then calls `_seed_initial_structures(village)` outside its lock block to place `LIBRARY`, `WORKSHOP`, and `FORGE` near the village center.

The result: expansion-founded outposts are structurally empty. Road segments from `create_road_between` appear at the site, but no buildings are placed. Agents that migrate into the outpost area have no structures to contribute work units to.

## Fix

### `src/hamlet/world_state/manager.py`

In `found_village`, after the `async with self._lock:` block closes (after the new village is created and queued for persistence), add:

```python
# Seed initial structures for the new outpost — same pattern as get_or_create_project
await self._seed_initial_structures(new_village)
```

Key constraint: call OUTSIDE the lock. `_seed_initial_structures` acquires the lock itself (via `create_structure`). Calling it inside the lock would deadlock because `asyncio.Lock` is not re-entrant.

Read `get_or_create_project` around line 219-226 for the exact pattern to follow. The structure is:
```python
async with self._lock:
    # create the village object and queue persistence
    ...
# Outside the lock:
await self._seed_initial_structures(new_village)
return new_village
```
