# WI-183 Notes: Daemon handling for new event types

## Context
WorldStateManager.handle_event() needs to handle 11 new HookType values without errors, and trigger visual activity for each.

## Mapping guidance (visual interest over accuracy)
SessionStart → get_or_create_project + get_or_create_session (entities exist for downstream)
SessionEnd → set all agents in session to AgentState.IDLE
SubagentStart → get_or_create_agent (spawn a new agent in the village)
SubagentStop → log only (agent will go idle naturally)
TeammateIdle → set matching agent to AgentState.IDLE (if found by teammate_name)
TaskCompleted → add_work_units(10) on a random structure in the project village
PostToolUseFailure → log only (error state visible in event log)
UserPromptSubmit → log only
PreCompact → log only
PostCompact → log only
StopFailure → log only

## Error handling
All state mutations wrapped in try/except with logger.warning per GP-7. Failure to trigger visual effect must never propagate.

## EventLogEntry summary format
Use event.hook_type.value as fallback. Specific formats:
"SessionStart: {event.source or 'startup'}"
"SessionEnd: {event.reason or ''}"
"SubagentStart: {event.agent_type or 'unknown'}"
"SubagentStop: {event.agent_type or 'unknown'}"
"TeammateIdle: {event.teammate_name or ''}"
"TaskCompleted: {event.task_subject or ''}"
"PostToolUseFailure: {event.tool_name or ''}"
"UserPromptSubmit" / "PreCompact" / "PostCompact" / "StopFailure"
