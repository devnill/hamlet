# WI-141: Fix hook registration end-to-end

## Context
Two hook registration paths exist, both broken:
1. `plugin.json` missing `hooks` field — hooks/hooks.json never loaded at plugin install
2. `install_hooks_to_settings` writes flat string format; Claude Code requires nested array format

Additionally, `${CLAUDE_PLUGIN_ROOT}` in hooks.json JSON values is not shell-expanded by Claude Code.

## Investigation Required
Before making changes, check the reference plugin implementations:
- `~/code/cyberbrain/.claude-plugin/plugin.json` — does it have a `hooks` field?
- `~/code/beepboop/.claude-plugin/plugin.json` — same check
- `~/code/ideate/.claude-plugin/plugin.json` — same check

Also check any existing `hooks/hooks.json` in those repos for the exact format.

## Expected Changes

### `.claude-plugin/plugin.json`
Add `hooks` field if the plugin schema supports it:
```json
{
  "hooks": "./hooks/hooks.json"
}
```
Or embed hooks inline if that's the correct format.

### `hooks/hooks.json`
If `${CLAUDE_PLUGIN_ROOT}` is not expanded in JSON values, the command paths need to use absolute paths or a different mechanism. Investigate whether the shell wrapper approach or direct Python invocation is correct.

### `src/hamlet/cli/commands/install.py`
Change `install_hooks_to_settings` to write the nested array format:
```python
settings.setdefault("hooks", {}).setdefault(hook_name, []).append({
    "hooks": [{"type": "command", "command": str(hook_path)}]
})
```
Or find the exact format by inspecting a real settings.json with hooks registered.

## Notes
- Read the reference plugins first — don't guess the format
- If the Claude Code plugin system supports a `hooks` field in plugin.json, use it and remove the manual `hamlet install` hook registration
- Document the verified format in a comment in the code
