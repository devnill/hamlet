## WI-138 Review

**Files reviewed:**
- `src/hamlet/simulation/animation.py`
- `src/hamlet/__main__.py`
- `src/hamlet/cli/commands/daemon.py`
- `src/hamlet/mcp_server/serializers.py`

### Verdict: FAIL

### Findings

- [SIGNIFICANT] `get_frames()` applies the spin-frame formula to all agents indiscriminately.

  **File:** `src/hamlet/simulation/animation.py:71-74`

  `_frames` stores raw ticks for both ACTIVE (spin) and ZOMBIE (pulse) agents — see `advance_frames()` at line 84, which increments `_frames[agent.id]` for both states. `get_frames()` computes every entry as `ticks // TICKS_PER_SPIN_FRAME % 4`, which is the spin formula. For zombie agents the correct formula is `ticks // TICKS_PER_PULSE_FRAME % 2`. Any remote viewer consuming `animation_frames` from the REST API will display the wrong frame index for zombie agents: the divisor is wrong (8 instead of 15) and the modulus is wrong (4 instead of 2).

  The spec says `get_frames()` must return "display frame indices". For zombies, the returned value does not represent a valid display frame index.

  Fix: record the agent state alongside the tick count, or maintain two separate maps, so `get_frames()` can apply the correct per-state formula:

  ```python
  def get_frames(self) -> dict[str, int]:
      result = {}
      for agent_id, ticks in self._frames.items():
          if agent_id in self._zombie_ids:
              result[agent_id] = (ticks // TICKS_PER_PULSE_FRAME) % 2
          else:
              result[agent_id] = (ticks // TICKS_PER_SPIN_FRAME) % 4
      return result
  ```

  Alternatively, store `(ticks, state)` tuples in `_frames` and branch on state in `get_frames()`.

- [MINOR] `serializers.py` — `serialize_state` accepts `animation_manager: Any = None` making the type invisible to static analysis. Given that `MCPServer` always passes a real `AnimationManager`, this parameter could be typed as `AnimationManager | None` with a proper import. Not a runtime defect, but a type-safety gap.

  **File:** `src/hamlet/mcp_server/serializers.py:93`

**Criteria check:**
- [x] MCPServer receives `animation_manager` in both `__main__.py` (line 100) and `daemon.py` (line 106). Both pass `animation_manager=animation_manager`. Confirmed.
- [x] `AnimationManager.get_frames()` exists at `animation.py:69`. It returns a `dict[str, int]` keyed by agent_id. It does not return `_frames` directly — computation is applied. However, as noted above, the computation is incorrect for zombie agents.
- [x] `serializers.py` calls `animation_manager.get_frames()` at line 99, not `_frames`. Confirmed.

## WI-140 Review

**File reviewed:** `src/hamlet/__main__.py`

### Verdict: PASS_WITH_NOTES

### Findings

- [MINOR] `main()` is annotated `-> None` but the `__main__` guard at line 258 passes its return value to `sys.exit`: `sys.exit(main())`. Since `main()` always calls `sys.exit()` internally and never returns normally, `sys.exit(main())` will receive `None`, which `sys.exit` treats as exit code 0. This means if a future refactor removes the internal `sys.exit` calls from `main()`, the outer guard will silently succeed regardless of the actual exit code. The annotation should be `-> int` (or `-> None` with the outer `sys.exit` removed), and the two should be consistent.

  **File:** `src/hamlet/__main__.py:242,257-258`

- [MINOR] Line 250 (`return`) is unreachable dead code — `sys.exit()` on line 249 always raises `SystemExit`.

  **File:** `src/hamlet/__main__.py:250`

**Criteria check:**

- [x] `sys.exit(exit_code if exit_code is not None else 0)` called at line 249.
- [x] When `_run_viewer()` returns 1, `sys.exit(1)` is called, producing a non-zero process exit. Confirmed by tracing the return paths in `_run_viewer()`.
