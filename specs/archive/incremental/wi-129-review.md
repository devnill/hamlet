### Findings

- [CRITICAL] `hooks.json` uses `${CLAUDE_PLUGIN_ROOT}` as a literal string in the `command` field. Claude Code reads `hooks.json` as JSON and passes the command string directly to the shell without performing environment variable expansion on JSON string values. The variable will not be substituted; the literal string `${CLAUDE_PLUGIN_ROOT}/hooks/pre_tool_use.sh` will be executed and will fail with "No such file or directory". The hooks must reference absolute paths or use a wrapper approach that the shell actually evaluates. One correct approach: register the hooks in `~/.claude/settings.json` or `.claude/settings.json` (where Claude Code reads its hook config) pointing at a thin loader script that is an absolute path and that itself expands `CLAUDE_PLUGIN_ROOT` — or register absolute paths directly in settings. The `hooks.json` file as a standalone deliverable has no defined load mechanism in the Claude Code hooks system; hooks must be in the Claude settings file at `~/.claude/settings.json` or `.claude/settings.json`. File: `/Users/dan/code/hamlet/hooks/hooks.json:8,19,30,41`

- [CRITICAL] There is no mechanism documented or implemented that loads `hooks/hooks.json` into Claude Code's active configuration. The Claude Code hooks system reads from `~/.claude/settings.json` or `.claude/settings.json` (project root), not from an arbitrary `hooks/hooks.json` file. As delivered, the hooks will never fire because Claude Code will never read this file. The work item does not address this integration gap. File: `/Users/dan/code/hamlet/hooks/hooks.json:1`

- [SIGNIFICANT] None of the four shell wrappers guard against `CLAUDE_PLUGIN_ROOT` being unset or empty. If the variable is unset, `exec "/hooks/pre_tool_use.py"` (with a leading slash but no meaningful path) is executed, which will fail silently or with an unhelpful error. Each script should validate the variable before exec: `[[ -z "${CLAUDE_PLUGIN_ROOT}" ]] && { echo "CLAUDE_PLUGIN_ROOT is not set" >&2; exit 1; }`. Files: `/Users/dan/code/hamlet/hooks/pre_tool_use.sh:2`, `/Users/dan/code/hamlet/hooks/post_tool_use.sh:2`, `/Users/dan/code/hamlet/hooks/notification.sh:2`, `/Users/dan/code/hamlet/hooks/stop.sh:2`

- [SIGNIFICANT] The shell wrappers do not forward stdin to the Python scripts. Claude Code delivers the hook payload as JSON on stdin. `exec` replaces the process image but stdin must be available; because none of the scripts explicitly pipe or redirect stdin, this will work correctly only if exec inherits the file descriptor — which it does in a normal shell. However, the scripts should be verified: `exec "${CLAUDE_PLUGIN_ROOT}/hooks/pre_tool_use.py" "$@"` should include `"$@"` to forward any arguments Claude Code passes on the command line. Currently all four scripts pass no arguments. The Stop hook is intended for cleanup on session termination; a 5-second timeout may be too short for cleanup tasks. The research document notes Stop hooks are best-effort. - [MINOR] The hooks.json outer structure wraps each hook type in an array with a single object that has only a `hooks` key and no `matcher` key. This matches the schema shown in the project's research document (an absent matcher means match-all), so it is structurally valid.
