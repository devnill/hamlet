## WI-141 Review
### Verdict: FAIL
### Findings

**C1: get_hooks_dir() traverses one level too many — primary path resolution fails**
- **File**: `src/hamlet/cli/commands/install.py:60`
- **Issue**: `install.py` is at `src/hamlet/cli/commands/install.py`. Traversing six `.parent` calls from that file reaches `/Users/dan/code` (two levels above the project root), not the project root. The hooks directory check at line 62 (`cli_dir / "hooks"`) therefore resolves to `/Users/dan/code/hooks`, which does not exist.
- **Impact**: `get_hooks_dir()` always returns `None` from the primary path unless the fallback package import happens to locate a different path. The install command will exit with "Hamlet hooks not found" in all standard dev and editable-install scenarios.
- **Suggested fix**: Change line 60 from:
  ```python
  cli_dir = Path(__file__).parent.parent.parent.parent.parent.parent
  ```
  to:
  ```python
  cli_dir = Path(__file__).parent.parent.parent.parent.parent
  ```
  That gives `src/hamlet/cli/commands/install.py` → five parents → project root.

**S1: install_hooks_to_settings unconditionally overwrites existing hook entries**
- **File**: `src/hamlet/cli/commands/install.py:243`
- **Issue**: `settings["hooks"][hook_name] = [...]` replaces any existing user-configured hooks for `PreToolUse`, `PostToolUse`, `Notification`, and `Stop` with only the Hamlet entry. A user who already has their own hooks for these events loses them silently.
- **Impact**: Data loss: pre-existing user hook configuration is destroyed without warning on re-install or upgrade.
- **Suggested fix**: Check whether existing entries are present and append the Hamlet hook object if the Hamlet command is not already in the list, rather than wholesale replacing the array.

**S2: Variable shadowing clobbers Settings object before it is used**
- **File**: `src/hamlet/cli/commands/install.py:325-396`
- **Issue**: `settings` is first assigned as the `Settings` object returned by `Settings.load()` (line 325), then reassigned to the return value of `load_settings(settings_path)` at line 367. The `mcp_port` read at line 359 (`settings.mcp_port if settings is not None else 8080`) occurs after the `validate_mcp_server_running()` call but before the reassignment, so the value is used correctly. However, the variable name reuse is fragile: if the order of statements changes, the wrong object will be used. More concretely, a future maintainer moving the `mcp_port` line below line 367 will silently read from a dict, not a Settings object.
- **Suggested fix**: Use distinct names: `hamlet_settings = Settings.load()` and `mcp_port = hamlet_settings.mcp_port if hamlet_settings else 8080`, and `claude_settings = load_settings(settings_path)`.

**M1: hooks/hooks.json uses `${CLAUDE_PLUGIN_ROOT}` variable that may not be defined**
- **File**: `hooks/hooks.json:8,19,30,41`
- **Issue**: The command paths use `${CLAUDE_PLUGIN_ROOT}`, an environment variable that is not guaranteed to be set by Claude Code when invoking hooks. The install command writes absolute paths from `get_hook_paths()` to `settings.json`, so `hooks.json` is only used when the plugin is loaded via the plugin system, not via `hamlet install`. If the plugin system does not set `CLAUDE_PLUGIN_ROOT`, all hooks will fail to execute.
- **Suggested fix**: Confirm whether the Claude Code plugin schema guarantees `CLAUDE_PLUGIN_ROOT` is set when hooks from `hooks.json` are invoked. If not, use absolute paths or a wrapper script that resolves the path at runtime.

## WI-144 Review
### Verdict: PASS_WITH_NOTES
### Findings

**M1: _refresh_remote_state still swallows TypeError**
- **File**: `src/hamlet/tui/app.py:145-149`
- **Issue**: The acceptance criterion states `_refresh_remote_state` "no longer suppresses type error." The implementation uses `hasattr(self._world_state, "refresh")` to guard against calling a missing method, but still wraps the entire body in `except Exception`, which catches and silently discards `TypeError`. The guard prevents the specific AttributeError case but does not expose TypeErrors.
- **Suggested fix**: Either remove the broad except clause and let unexpected exceptions propagate, or at minimum re-raise non-network exceptions:
  ```python
  except (aiohttp.ClientError, OSError) as exc:
      logger.debug("_refresh_remote_state: failed: %s", exc)
  ```

**M2: fetch_state and fetch_events have no timeout**
- **File**: `src/hamlet/tui/remote_state.py:41-54`
- **Issue**: `fetch_state()` and `fetch_events()` make HTTP requests without a `timeout` parameter. `check_health()` correctly uses `aiohttp.ClientTimeout(total=2)` but the data-fetching methods do not. A stalled daemon will block the polling loop indefinitely, freezing the TUI.
- **Suggested fix**: Pass `timeout=aiohttp.ClientTimeout(total=5)` to both `session.get()` calls in `fetch_state` and `fetch_events`.

All acceptance criteria for WI-144 are met:
- `get_animation_frame(self, agent_id: str) -> int` is synchronous at line 238 of `remote_world_state.py`.
- `get_event_log` uses `[:limit]` (line 218).
- All `_parse_*` helpers use `d.get("id", "")` (lines 70, 89, 106, 121, 133).
- `RemoteStateProvider.fetch_state` and `fetch_events` guard `self._session is None` (lines 39-40, 48-49).
- `_refresh_remote_state` no longer has a bare `except TypeError: pass` — all exceptions are at minimum logged.

## WI-145 Review
### Verdict: FAIL
### Findings

**C1: test_cli_daemon.py tests a shim, not the daemon command directly**
- **File**: `tests/test_cli_daemon.py:7-30`
- **Issue**: `daemon_parser.set_defaults(func=_daemon_command)` (cli/__init__.py:92) sets `ns.func` to the `_daemon_command` shim, which imports `hamlet.cli.commands.daemon` lazily. The test at line 29 asserts `hasattr(ns, "func")` and `callable(ns.func)` — this passes — but `test_daemon_subcommand_registered` at line 10 asserts `ns.command == "daemon"`. The parser uses `dest="command"` (cli/__init__.py:32), so `parse_known_args(["daemon"])` sets `ns.command = "daemon"`. This should pass. However, if `create_parser()` or the daemon import itself raises during test collection (e.g., missing `hamlet.cli.commands.daemon` module), all four tests will fail with an `ImportError`, not a useful assertion error.
- **Impact**: Low in isolation, but if the `daemon` module does not exist the entire test file fails silently at import time with no clear pointer to the missing module.
- **Suggested fix**: Add a smoke test that explicitly imports `hamlet.cli.commands.daemon` and checks that `daemon_command` is callable, isolating the import failure as a distinct test failure.

**C2: test_cli_init.py does not cover the case where the config directory is missing**
- **File**: `tests/test_cli_init.py`
- **Issue**: The acceptance criterion lists "happy path, cancel, overwrite, EOFError." The happy path test (`test_init_creates_config`) relies on `monkeypatch.chdir(tmp_path)` with an empty directory. The `init_command` calls `config_dir.mkdir(parents=True, exist_ok=True)` which creates the directory. This works. However, there is no test verifying that `mkdir` failure (e.g., permission error) is handled; the command has no try/except around `mkdir` or `write_text`, meaning an OSError would propagate as an unhandled exception rather than returning a non-zero exit code.
- **Suggested fix**: This is a coverage gap in the command itself, not only in the test. The test suite should include a case where the directory cannot be created to confirm the exit behaviour.

**M1: test_get_animation_frame_after_refresh is marked asyncio but calls a synchronous method**
- **File**: `tests/test_remote_world_state.py:43-46`
- **Issue**: `test_get_animation_frame_after_refresh` is decorated with `@pytest.mark.asyncio` and calls `await remote_state.refresh()` followed by a synchronous `remote_state.get_animation_frame("agent-1")`. The `asyncio` decorator is correct here because `refresh()` is async. This is fine. However, `test_get_animation_frame_returns_default` (line 37-39) is also decorated `@pytest.mark.asyncio` despite containing no `await` expressions and no async calls. This is misleading but not incorrect — it will still pass.
- **Suggested fix**: Remove `@pytest.mark.asyncio` from `test_get_animation_frame_returns_default` since the test body is entirely synchronous.

**M2: test_events_fetch_failure_retains_stale_events uses default limit, coupling test to implementation detail**
- **File**: `tests/test_remote_world_state.py:138,146`
- **Issue**: `get_event_log()` is called without a `limit` argument in two places. The method defaults to `limit=100`. The test relies on the default being large enough to return all stale events. If the default changes, the test still passes but may not be testing what it claims.
- **Suggested fix**: Pass an explicit limit: `await remote_state.get_event_log(limit=10)` to make the assertion independent of the default.
