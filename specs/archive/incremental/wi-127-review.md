- [SIGNIFICANT] `_refresh_remote_state` calls `self._world_state.refresh()` but the refresh loop should only fire in viewer mode. The condition check on line 129 of `app.py` correctly gates the interval registration on `remote_provider is not None`, but when `remote_provider` is `None` (normal daemon mode) and `_world_state` is a real `WorldStateManager`, a stray call to `_world_state.refresh()` would raise `AttributeError` since `WorldStateManager` has no `refresh()` method. The interval is not registered in daemon mode so this cannot currently be triggered at runtime, but it is one line away from a crash if the guard condition ever changes or is misread. The method should call `self._remote_provider.refresh()` or, better, call `await self._world_state.refresh()` only after asserting `isinstance(self._world_state, RemoteWorldState)`. As written, the `# type: ignore[attr-defined]` suppresses the type-checker warning that would catch this. File: `/Users/dan/code/hamlet/src/hamlet/tui/app.py:135`

- [SIGNIFICANT] `get_animation_frame` signature diverges from the spec. The spec (`specs/plan/notes/127.md:90`) defines `def get_animation_frame(self, agent_id: str) -> int` — a synchronous method that takes `agent_id` and looks up per-agent animation frames from `_cached_state.get("animation_frames", {})`. The implementation (`remote_world_state.py:236`) is `async def get_animation_frame(self) -> int` with no `agent_id` parameter and always returns `0`, discarding any animation frame data the daemon may provide. If any widget calls `get_animation_frame(agent_id)` it will receive a `TypeError`. File: `/Users/dan/code/hamlet/src/hamlet/tui/remote_world_state.py:236`

- [SIGNIFICANT] `get_event_log` slices with `self._event_log[-limit:]` (most recent N entries) but the spec pseudocode (`notes/127.md:78`) uses `self._cached_events[:limit]` (first N entries). These return opposite ends of the list. Whether "most recent" or "oldest" is semantically correct depends on ordering from the daemon, but the implementation silently changed the slicing direction from the spec without documenting it as a deviation. File: `/Users/dan/code/hamlet/src/hamlet/tui/remote_world_state.py:216`

- [CRITICAL] `check_health`, `fetch_state`, and `fetch_events` all call `self._session.get(...)` with no guard for `self._session is None`. If `check_health` is called before `start()` (or after `stop()`), the `type: ignore[union-attr]` suppresses the type error but the runtime will raise `AttributeError: 'NoneType' object has no attribute 'get'`. In `_run_viewer`, `check_health` is called immediately after `provider.start()`, so the normal path is safe, but there is nothing preventing misuse and the error message will be cryptic. The `# type: ignore` annotations mask the hazard. A guard (`if self._session is None: raise RuntimeError(...)`) should be added to all three methods. File: `/Users/dan/code/hamlet/src/hamlet/tui/remote_state.py:27,37,44`

- [MINOR] `RemoteWorldState.refresh` fetches state and events in two separate try/except blocks. If `fetch_state` fails, the agent/structure/village/project caches are left at their previous values while `_event_log` may still be updated by the second fetch. This is intentional stale-cache behavior per GP-7, but the two independent catches mean a partial update is possible on every failure: old world state with new events, or vice versa. The spec pseudocode (`notes/127.md:60-63`) performs both fetches in a single try block to keep the cache consistent. File: `/Users/dan/code/hamlet/src/hamlet/tui/remote_world_state.py:185-192`
